# 校园食堂就餐仿真业务逻辑技术文档 v1.0

## 0. 文档用途

本文档用于定义校园食堂就餐仿真系统的业务逻辑边界、对象模型、状态机、事件流、决策规则、统计指标和阶段性交付范围。

后续将基于本文档继续拆分三类 TODO：

- 前端 TODO：配置入口、画布展示、状态可视化、统计面板、联调字段展示。
- 后端 TODO：对象模型、状态机、事件流、仿真推进、路径规划、实时数据输出。
- 数据处理 TODO：事件记录、指标计算、实验结果导出、统计口径校验。

本文档中的“分阶段”只表示实现优先级，不引入临时业务设计。所有阶段都应朝同一套最终业务模型演进。

相关文档：

- `docs/frontend_backend_integration_test_plan.md`：联调前后端独立测试方案与联调流程。
- `docs/realtime_frame_interface.md`：仿真实时帧 `frame` 接口契约。

## 3. 仿真逻辑概览

目标主流程如下：

```text
学生入场
  -> 思考
  -> 选择菜品 / 窗口
  -> 前往窗口
  -> 排队
  -> 完成出餐
  -> 找座 / 等座
  -> 前往座位
  -> 就餐
  -> 前往碗筷收集处
  -> 前往出口
  -> 离场
```

当前代码已经覆盖的主流程：

```text
学生入场
  -> 思考
  -> 选择窗口
  -> 前往窗口
  -> 排队
  -> 完成出餐
  -> 找座 / 等座
  -> 前往座位
  -> 就餐
  -> 前往碗筷收集处
  -> 前往出口
  -> 离场
```

当前缺口：

- 缺少菜品级选择、价格、库存、售罄和订单对象。
- 配置弹窗中的 `max_active_students` 仍与学生总数绑定。
- 缺少“到达碗筷收集处”的事件级记录。
- 空间布局仍写死在仿真引擎中。
- 多入口、多出口、多桌型、全平面通行还未实现。

## 4. 核心对象模型

### 4.1 学生 `Student`

学生是仿真系统中的主动行为对象。

| 字段 / 逻辑 | 状态 | 说明 |
| --- | --- | --- |
| `id` | `done` | 学生编号 |
| `spawn_time` | `done` | 学生进入系统的仿真时间 |
| `x, y` | `done` | 学生当前位置坐标 |
| `state` | `done` | 学生当前状态 |
| `meat_pref, veg_pref` | `done` `todel` | 当前菜品偏好，范围 `0.0 ~ 1.0`，后续替换 |
| 多维偏好向量 | `todo` | 口味、价格敏感度、荤素偏好、等待容忍度等 |
| `hesitation_time` | `done` | 思考时长，范围 `10.0 ~ 28.0` |
| `appetite` | `done` | 食量，范围 `0.8 ~ 1.8` |
| `eat_speed` | `done` | 进食速度，范围 `0.028 ~ 0.075` |
| `appetite / eat_speed` | `done` | 当前就餐时长计算方式 |
| `walk_speed` | `done` | 普通移动速度，范围 `8.0 ~ 14.0` |
| `table_walk_time` | `done` | 去座位预期耗时，范围 `35.0 ~ 70.0` |
| `stall_id` | `done` | 目标窗口编号 |
| `table_id` | `done` | 目标餐桌编号 |
| `seat_index` | `done` | 目标座位编号 |
| `group_id` / 同行关系 | `todo` | 独行、两人结伴、多人同行 |

学生偏好规则：

- 当前使用连续值，不是“偏荤 / 偏素”文字标签。`done` `todel`
- 学生偏好直接映射到菜品特征向量。`todo`
- 学生先给候选菜品打分，再选择提供高分菜品的窗口。`todo`

同行关系规则：

- 是否有同行状态。`todo`
- 独行。`todo`
- 两人结伴。`todo`
- 多人同行。`todo`
- 是否需要随机生成同行关系。`todo`
- 学生生成时按比例生成同行组；同组成员共享 `group_id`，优先统一选窗口、统一找座、优先同桌或相邻座位。`todo`

### 4.2 窗口 `Stall`

窗口负责售卖菜品、维护队列、处理订单和输出营业状态。

| 字段 / 逻辑 | 状态 | 说明 |
| --- | --- | --- |
| `id` | `done` | 窗口编号 |
| `x, y` | `done` | 窗口位置坐标 |
| `meat_ratio, veg_ratio` | `done` `todel` | 当前窗口偏好特征，后续替换 |
| 多维菜品特征 | `todo` | 和学生偏好匹配 |
| `cook_time` | `done` `todel` | 当前单份出餐时间，范围 `12.0 ~ 28.0` |
| 菜品级出餐时间 | `todo` | 每个菜品对应不同出餐时间 |
| `queue` | `done` `todel` | 当前窗口队列；主流程保留单队列 |
| `ready_times` | `done` `todel` | 当前预约出餐时间表 |
| 订单对象列表 | `todo` | 记录菜品、创建时间、开始处理时间、完成时间 |
| 窗口状态字段 | `todo` | 待营业、营业中、已售罄 |
| 菜品表 | `todo` | 窗口出售内容 |
| 菜品价格 | `todo` | 每个菜品的价格 |
| 菜品余量 | `todo` | 每个菜品的库存 |

窗口队列规则：

- 某些窗口可以拥有两个队列：付款队列，付完钱就走。`todo`
- 某些窗口可以拥有两个队列：等饭队列，拿到饭再走。`todo`
- 本版主流程保留单队列，支付和出餐视为一个连续服务过程；双队列移入扩展设计。`todel`

窗口出餐规则：

- 订单完成时间 = 队列等待时间 + 菜品基础出餐时间。`todo`
- 窗口平均速度只作为整体修正，不再替代菜品时间。`todo`
- 出餐完成时同步扣减菜品库存。`todo`
- 若库存扣减后为零，则触发窗口售罄状态更新。`todo`

窗口状态规则：

- 当前没有单独的窗口状态字段。`doing`
- 当前只能通过队列长度和出餐队列推导。`doing`
- 待营业。`todo`
- 营业中。`todo`
- 已售罄。`todo`
- 营业状态由“营业时段 + 是否仍有可售菜品”共同决定。`todo`
- 核心菜品库存为零时切换为“已售罄”。`todo`
- 当前没有正式拥堵阈值定义。`todo`
- 当 `queue_count >= 8` 时判定为拥堵。`todo`

### 4.3 餐桌 / 座位 `Table / Seat`

餐桌和座位负责承载就餐行为，并影响路径规划、空间碰撞和利用率统计。

| 字段 / 逻辑 | 状态 | 说明 |
| --- | --- | --- |
| `id` | `done` | 餐桌编号 |
| `x, y` | `done` | 餐桌位置坐标 |
| `seats` | `done` `todel` | 当前每张桌子固定 `4` 个座位 |
| 2 人桌、6 人桌 | `todo` | 四人桌仍为主体 |
| `table_type` | `todo` | 餐桌类型 |
| `seat_count` | `todo` | 座位数量 |
| `seat is None` | `done` | 空闲 |
| `seat == student.id` | `done` `todel` | 旧兼容字段中表示已预留或占用，无法区分具体状态 |
| 独立座位状态字段 | `done` | 当前使用 `Seat.status` |
| `reserved` | `done` | 预留状态 |
| 餐桌碰撞箱 | `todo` | 参与障碍物生成和路径规划 |

座位规则：

- 当前已有单独的座位状态字段。`done`
- 增加 `reserved` 状态，分配后未到达记为预留，到达座位后切换为占用。`done`
- `tables[].seats` 仍保留旧兼容输出，无法区分 `reserved` 和 `occupied`。`doing`
- `tables[].seat_frames` 输出 `free`、`reserved`、`occupied`。`done`
- 保留“预留”状态，先锁定座位，再在学生到达后切换为占用。`done`
- 单人优先分配 2 人桌或剩余空位较少的桌子。`todo`
- 多人同行优先分配能容纳整组的桌型。`todo`

| `exit = (1228.0, 740.0)` | `done` `todel` | 当前单出口 |

## 11. 逻辑变更检查标准

每次修改业务逻辑前，需要回答以下问题：

1. 谁会变？
   - 学生。
   - 窗口。
   - 座位。
   - 空间。
   - 统计模块。

2. 什么时候变？
   - 是什么事件触发了变化。

3. 变成什么？
   - 变化后的状态是什么。

4. 会影响谁？
   - 一个对象变化后，是否带动其他对象一起变化。

示例：

- 学生开始就餐时：
  - 学生状态变为 `EATING`。
  - 座位状态变为占用。
  - 统计模块记录开始就餐时间。

## 12. 当前推荐实现顺序

P0 主流程、座位状态、核心事件记录、基础统计和前端展示已经基本落地。当前建议按以下顺序继续推进：

1. 配置：让配置弹窗中的 `max_active_students` 成为独立输入，不再传 `max(55, total_students)`。
2. 事件：补充“到达碗筷收集处”事件，使“结束就餐 -> 回收点 -> 出口”可由事件表审计。
3. 契约：决定是否继续保留 `tables[].seats` 旧字段；若保留，文档中明确它只作兼容，不作为状态判断来源。

6. 行为：引入多维偏好、参数化权重、价格权重和随机扰动。
7. 群体：引入同行关系和多桌型。
