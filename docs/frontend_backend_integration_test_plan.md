# 前后端联调与独立测试方案

## 1. 文档目标

本文档定义前端、后端在正式联调前如何分别调试，以及联调时如何保证双方按同一套仿真实时数据契约对齐。

接口字段契约见：

- `docs/realtime_frame_interface.md`

业务逻辑和阶段性 TODO 见：

- `docs/business_logic_draft.md`

## 2. 基本原则

- 前端、后端都以 `frame` 数据契约为对齐边界。
- 前端不依赖后端内部类名，只依赖接口字段。
- 后端不依赖前端绘制细节，只保证状态、事件、统计和空间数据输出稳定。
- 示例数据必须使用最终字段名，不因阶段性交付引入临时字段。
- 固定 `seed` 的仿真结果应可复现，方便联调复盘。

## 3. 后端独立调试方案

后端需要在没有真实前端依赖的情况下验证仿真逻辑。

建议方式：

- 以 `SimulationConfig` 为输入，固定 `seed`，生成可复现仿真。
- 每次仿真 tick 输出统一 `frame` 字典。
- 本地检查 `frame` 是否满足 `docs/realtime_frame_interface.md`。
- 本地检查状态枚举是否只输出约定值。
- 本地检查事件记录是否能覆盖统计指标。

后端自测关注点：

- 学生状态是否按主流程推进。
- 队列长度是否和学生状态一致。
- 座位空闲 / 预留 / 占用是否一致。
- 碗筷收集处是否成为离场前必经事件。
- `served_students`、`active_students`、事件记录是否互相对得上。
- 固定 `seed` 后，关键指标是否可复现。

## 4. 前端独立调试方案

前端需要在后端未完成全部逻辑前，用稳定数据结构调试展示。

建议方式：

- 使用固定 shape 的示例 `frame` 数据调试画布和统计面板。
- 示例数据必须覆盖学生、窗口、餐桌、座位、空间、统计六类对象。
- 前端对未知枚举值做兜底展示。
- 前端不能吞掉必需字段缺失错误。
- 当前阶段尚未实现的字段可以为空数组或 `null`，但字段名需要保持稳定。

前端自测关注点：

- 学生不同状态是否有可区分展示。
- 窗口待营业 / 营业中 / 已售罄 / 拥堵是否能展示。
- 座位空闲 / 预留 / 占用是否能展示。
- 多入口、多出口、碗筷收集点字段即使当前为空，也能兼容。
- 统计面板能展示基础指标和缺省值。
- 路径调试显示不会因为 `path` 为空数组而异常。

## 5. 联调流程

建议按以下顺序联调：

1. 后端先输出最小 `frame`，前端只验证字段读取和画布刷新。
2. 对齐学生状态枚举，确认每个状态都有展示策略。
3. 对齐座位状态枚举，确认预留和占用不会混淆。
4. 对齐窗口状态枚举，确认售罄和拥堵能同时表达。
5. 接入碗筷收集处，确认路径和状态推进一致。
6. 接入统计字段，确认实时值和事件累计值口径一致。
7. 固定 `seed` 做一轮可复现联调，记录关键时刻的 `frame`。

## 6. 联调验收口径

一次联调是否通过，至少看以下条件：

- 前端没有字段缺失导致的异常。
- 后端没有输出前端未约定的状态值。
- 同一个学生不会同时处于两个业务状态。
- 一个座位不会同时被多个学生占用或预留。
- 学生离场前必须完成碗筷收集处事件。
- 统计中的“已完成就餐人数”与离场事件口径一致。
- 固定随机种子下，关键指标可复现。

## 7. 推荐联调用例

### 7.1 基础流转

目的：验证学生从入场到离场的完整主流程。

检查点：

- 学生进入 `DECIDING`。
- 学生进入 `MOVING_TO_QUEUE`。
- 学生进入 `QUEUED`。
- 学生完成出餐。
- 学生进入找座 / 前往座位。
- 学生进入 `EATING`。
- 学生前往碗筷收集处。
- 学生进入 `LEAVING`。
- 学生最终进入 `DONE` 并从活跃列表移除。

### 7.2 无空座等待

目的：验证无空座时的等待状态和重试逻辑。

检查点：

- 没有空座时学生进入 `WAITING_SEAT`。
- 有座位释放后学生重新进入找座流程。
- 座位预留和占用状态不混淆。

### 7.3 窗口队列

目的：验证窗口排队、出餐和队列长度统计。

检查点：

- `queue_count` 和窗口队列人数一致。
- 队首优先完成出餐。
- `queue_count >= 8` 时可标记拥堵。

### 7.4 统计口径

目的：验证实时指标和事件累计指标一致。

检查点：

- `active_students` 等于未完成离场的学生数量。
- `spawned_students` 不小于活跃学生数量。
- 平均等待时间来源于“进入窗口队列 -> 完成出餐”。
- 平均就餐总耗时来源于“学生入场 -> 学生离场”。

## 8. 问题定位建议

联调中出现问题时，先按以下顺序定位：

1. 字段是否存在。
2. 字段类型是否符合接口文档。
3. 状态枚举是否符合接口文档。
4. 同一对象状态是否自洽。
5. 事件顺序是否符合业务逻辑文档。
6. 统计值是否能从事件记录推导。
