# 项目开发规范

## 1. 文档目的

本文档定义校园食堂就餐仿真系统的日常开发规范，用于统一代码组织、接口边界、测试方式、文档维护和提交前检查。

本规范适用于以下场景：

- 新增仿真业务逻辑。
- 修改前端画布、统计面板或配置入口。
- 调整实时 `frame` 接口、事件记录或统计口径。
- 编写测试脚本、联调用例和阶段性 TODO。
- 进行代码审查、合并前自检和问题定位。

相关文档：

- `README.md`：项目简介与基础运行说明。
- `docs/business_logic_draft.md`：业务逻辑、对象模型、状态机和阶段性 TODO。
- `docs/realtime_frame_interface.md`：后端输出给前端的实时 `frame` 数据契约。
- `docs/frontend_backend_integration_test_plan.md`：前后端独立测试与联调方案。

## 2. 项目技术栈与运行环境

### 2.1 基础环境

- Python 版本：`3.10`。
- GUI 框架：`PyQt6>=6.7,<7`。
- 当前测试框架：标准库 `unittest`。
- 入口文件：`main.py`。

### 2.2 推荐本地启动流程

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python main.py
```

Windows PowerShell 可使用：

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
python main.py
```

### 2.3 依赖管理

- 运行依赖统一写入 `requirements.txt`。
- 不要把虚拟环境、IDE 配置、缓存、日志和本地 `.env` 提交到仓库。
- 新增第三方依赖前，需要确认标准库或现有依赖不能满足需求。
- GUI、仿真、测试相关依赖要避免版本过宽；有兼容风险时写明上限。

## 3. 目录职责

当前项目采用接近 MVC 的分层组织。新增代码应优先放入既有目录，而不是在根目录堆积文件。

| 路径 | 职责 |
| --- | --- |
| `main.py` | 应用入口，只负责创建 `QApplication`、窗口和控制器。 |
| `models/` | 仿真模型、实体、事件记录、统计计算和核心业务规则。 |
| `controllers/` | 连接 UI 信号与仿真 worker，管理线程生命周期。 |
| `views/` | PyQt 界面、画布、配置弹窗和统计面板。 |
| `utils/` | 字体、数学、通用帮助函数等无业务状态工具。 |
| `config/` | 默认配置、可配置参数或后续空间布局配置。 |
| `tests/` | 自动化单元测试。 |
| `scripts/` | 自测、联调、数据验证等开发辅助脚本。 |
| `docs/` | 技术文档、接口契约、开发规范和阶段计划。 |

### 3.1 分层边界

- `models/` 不应依赖 `views/` 的绘制细节。
- `views/` 不应直接修改仿真内部对象，只消费 `frame` 字典和公开信号。
- `controllers/` 负责信号连接、线程启动、暂停、停止和清理，不承载业务计算。
- `utils/` 中的函数应保持无副作用，避免偷偷读取或修改全局状态。
- `main.py` 保持轻量，不放业务逻辑、样式逻辑或测试逻辑。

## 4. Python 代码规范

### 4.1 语言与类型

- 新文件统一添加 `from __future__ import annotations`。
- 业务数据优先使用 `dataclass`、`Enum` 和明确的类型标注。
- 枚举值使用稳定的字符串值，避免前端依赖 Python 枚举名。
- 函数参数和返回值应尽量补齐类型标注。
- 可以为 UI 回调保留 `object` 类型，但进入业务层前要转换成明确结构。

### 4.2 命名规范

- 文件名、模块名：`snake_case.py`。
- 函数、变量、方法：`snake_case`。
- 类名、数据类、枚举类：`PascalCase`。
- 常量：`UPPER_SNAKE_CASE`。
- 私有辅助函数或内部状态：以 `_` 开头，例如 `_build_frame()`。

### 4.3 代码组织

- 一个函数只做一类事情；复杂流程拆成可命名的私有方法。
- 修改状态机时，要让状态变更、事件记录、目标点设置处于同一个清晰流程中。
- 不要在视图层硬编码后端业务规则；视图只做展示、交互和缺省兜底。
- 不要在模型层硬编码 UI 文案、颜色或控件行为。
- 复杂公式、统计口径、路径规划策略需要用短注释说明原因。

### 4.4 异常处理

- 后端 worker 中的运行时异常应通过 `errorOccurred` 信号交给 UI 展示。
- 不要用裸 `except` 静默吞掉错误。
- 数据校验失败时，应尽量记录可定位的信息，例如学生编号、事件类型、状态名。
- 对用户输入配置做边界保护，例如数量最小值、时间倍率范围、空列表兜底。

## 5. 仿真模型开发规范

### 5.1 状态机

学生状态以 `models/entities.py` 中的 `StudentState` 为准。新增或调整状态时必须同步：

- `StudentState` 枚举。
- `SimulationWorker` 中的状态推进逻辑。
- `docs/business_logic_draft.md` 的业务流程说明。
- `docs/realtime_frame_interface.md` 的 `StudentFrame` 状态枚举。
- `views/canvas_widget.py` 和 `views/stats_panel.py` 中相关展示逻辑。
- 覆盖状态流转的测试或自测脚本。

状态推进应遵守以下原则：

- 同一学生同一时刻只能处于一个业务状态。
- 状态变更前后要能从事件记录或 `frame` 中复盘。
- 离场前的业务节点不能被 UI 计数逻辑误判为已完成。
- 如果状态用于兼容旧接口，要在接口文档中写明兼容期限和替代字段。

### 5.2 时间与随机数

- 仿真内部统一使用游戏时间 `game_time`，单位为秒。
- 现实时间只用于 worker tick 和 `time_scale` 换算，不应进入业务统计口径。
- 所有随机行为通过 `SimulationWorker.rng` 生成，保证固定 `seed` 可复现。
- 不要在核心仿真流程中直接调用全局 `random`。

### 5.3 坐标与空间

- 坐标采用画布坐标系，字段名统一为 `x`、`y`。
- 当前布局尺寸以 `SimulationWorker.width`、`SimulationWorker.height` 为准。
- 后续空间配置应优先抽到 `config/`，不要继续扩大硬编码布局。
- 新增空间对象时，应同时考虑绘制、碰撞、路径和 `frame` 输出。

### 5.4 座位与窗口

- 座位状态以 `SeatStatus` 为准：`free`、`reserved`、`occupied`。
- 分配座位后但未到达座位前，应使用 `reserved`，不要提前记为 `occupied`。
- 窗口队列变更要和 `queue_count`、事件记录、统计样本保持一致。
- 后续菜品、订单、库存、售罄逻辑应以 `docs/business_logic_draft.md` 的 P1 规划为准。

## 6. 实时 `frame` 接口规范

前后端以 `frame` 字典作为边界。后端可以重构内部类，但不能随意破坏 `frame` 字段。

### 6.1 基本原则

- 字段契约以 `docs/realtime_frame_interface.md` 为准。
- 新字段优先使用最终字段名，不为短期联调创造临时字段。
- 必需字段缺失属于后端问题，不应让前端静默忽略。
- 未实现的计划字段可以输出空数组、`null` 或合理默认值，但字段类型要稳定。
- 字段改名或删除必须先更新接口文档，并说明兼容策略。

### 6.2 输出要求

`frame` 顶层应保持以下信息清晰可读：

- 仿真时间：`game_time`、`duration`、`time_scale`。
- 人数统计：`spawned_students`、`served_students`、`active_students`。
- 空间信息：入口、出口、路径、窗口、餐桌、碗筷收集点。
- 实体信息：学生、窗口、餐桌、座位。
- 统计信息：`stats`。

### 6.3 前端消费要求

- 前端只依赖 `frame` 字段，不直接依赖 `SimulationWorker` 内部属性。
- 前端应对未知状态有兜底展示，但不能掩盖必需字段缺失。
- `path`、`students`、`stalls`、`tables` 等数组为空时，画布不能报错。
- 统计字段为 `None` 时，应显示空值、占位或“暂无”，不要显示误导性 `0`。

## 7. 事件记录与统计规范

### 7.1 事件记录

事件记录由 `models/data_recorder.py` 负责，P0 事件类型以 `P0_EVENT_TYPES` 为准：

- `student_spawned`
- `queue_started`
- `food_ready`
- `eating_started`
- `eating_finished`
- `student_left`

新增事件类型时必须同步：

- `P0_EVENT_TYPES` 或后续阶段事件集合。
- `EventRecordP0` 或新的事件数据结构。
- 统计推导逻辑。
- 单元测试。
- `docs/business_logic_draft.md` 和 `docs/realtime_frame_interface.md`。

### 7.2 统计口径

当前基础统计口径：

- 平均等待时间：从 `queue_started` 到 `food_ready`。
- 平均就餐总耗时：从 `student_spawned` 到 `student_left`。
- 最大在场人数：由入场、离场事件推导。
- 窗口最大队列长度：优先使用队列采样，缺失时由事件推导。
- 座位利用率：座位被占用总时长 / 总座位时长。

统计开发要求：

- 统计值必须能从事件或采样数据复盘。
- 缺少配对事件时返回 `None` 或跳过该样本，不要制造伪数据。
- 负时长、未知事件类型等异常情况应记录到 `DataRecorder.issues`。
- 实时统计和最终统计要使用同一套口径。

## 8. PyQt 前端开发规范

### 8.1 线程边界

- 仿真运行在 `QThread` 中的 `SimulationWorker`。
- UI 更新必须通过信号回到主线程。
- 不要在 UI 线程中执行耗时仿真循环、长时间睡眠或大规模计算。
- 停止、暂停、时间倍率调整通过 controller 信号传递给 worker。

### 8.2 视图职责

- `MainWindow` 负责顶层布局、工具栏、状态展示和信号转发。
- `CanvasWidget` 负责画布绘制和可视化调试层。
- `StatsPanel` 负责统计面板展示。
- `ConfigDialog` 负责用户输入并生成 `SimulationConfig`。

### 8.3 UI 文案和样式

- 用户可见文案保持中文，并尽量使用业务语言。
- 样式集中在对应视图组件中，避免散落到控制器或模型。
- 新增状态颜色时要考虑和现有状态区分，并提供未知状态兜底。
- 控件禁用、暂停、停止、结束等状态必须和实际 worker 状态一致。

## 9. 测试规范

### 9.1 必跑检查

修改模型、数据记录或统计逻辑后，至少运行：

```bash
python -m unittest discover -s tests
```

涉及后端主流程时，建议额外运行：

```bash
python scripts/backend_self_test.py
python scripts/test_data_recorder_p0.py
```

涉及 UI 时，至少手动启动一次：

```bash
python main.py
```

### 9.2 测试编写原则

- 新增统计口径必须有单元测试。
- 新增状态流转必须有可复现用例，优先固定 `seed`。
- 测试应断言业务结果，而不是断言实现细节。
- 对异常输入、缺失事件、空列表、边界数量要有覆盖。
- UI 变更如果难以自动化，需在提交说明或文档中写清手动验证点。

### 9.3 自测数据

- 自测脚本放在 `scripts/`。
- 自动化测试放在 `tests/`。
- 示例 `frame` 数据应使用最终字段名，并覆盖空数组、`None`、未知状态兜底。

## 10. 文档维护规范

文档是开发的一部分。以下变更必须同步文档：

- 改状态机：更新 `docs/business_logic_draft.md` 和 `docs/realtime_frame_interface.md`。
- 改 `frame` 字段：更新 `docs/realtime_frame_interface.md`。
- 改联调流程或验收口径：更新 `docs/frontend_backend_integration_test_plan.md`。
- 改统计口径：更新业务文档、接口文档和测试说明。
- 改开发流程、目录职责或提交检查：更新本文档。

文档写作要求：

- 使用中文说明业务意图，代码字段保持英文原名。
- 新旧接口兼容时明确标注当前字段、计划字段和迁移策略。
- TODO 需要写清阶段、目的和验收口径，不只写“优化”。
- 文档中的命令必须能在项目根目录执行。

## 11. 配置与数据规范

- 仿真参数统一通过 `SimulationConfig` 或配置文件进入系统。
- `SimulationConfig` 增加字段后，要同步 `ConfigDialog`、默认值、文档和测试。
- 配置字段必须有明确单位，例如秒、分钟、人数、倍率、像素坐标。
- JSON 配置应保持可读缩进，避免提交本地临时实验参数。
- 固定随机种子的实验配置应记录 `seed`，方便复现。

## 12. Git 与提交规范

### 12.1 提交前自检

提交前至少确认：

- 代码可以运行或相关测试通过。
- 没有提交 `.venv/`、缓存、日志、本地环境文件。
- 接口、状态、统计、文档变更已经同步。
- 没有无关格式化或大范围重排。
- 没有把调试打印、临时路径、个人机器配置留在代码里。

### 12.2 建议提交信息

提交信息建议使用简短中文或英文动词开头：

```text
补充 P0 数据记录统计测试
修正学生找座状态流转
更新实时 frame 接口文档
Add backend self-test for queue metrics
```

一次提交应聚焦一个主题。不要把业务逻辑、UI 重构、文档整理和格式化混在同一个提交里。

## 13. 代码审查清单

评审时优先看行为正确性和接口风险：

- 状态流转是否符合业务主流程。
- `frame` 输出是否和接口文档一致。
- 事件记录是否能推导统计结果。
- 固定 `seed` 下结果是否可复现。
- UI 是否只依赖公开字段。
- worker 生命周期是否能正常停止、暂停和清理。
- 新增配置是否有默认值和边界校验。
- 新增逻辑是否有测试或明确手动验证说明。
- 文档是否同步更新。

## 14. 常见开发场景规范

### 14.1 新增学生状态

1. 在 `StudentState` 中新增稳定字符串枚举。
2. 在 `SimulationWorker` 中实现进入、更新和退出逻辑。
3. 在状态变更处记录必要事件。
4. 在 `_build_frame()` 中输出该状态。
5. 在前端画布和统计面板中增加展示或兜底。
6. 更新接口文档和业务文档。
7. 补充测试或自测脚本。

### 14.2 新增统计指标

1. 明确指标定义、单位和是否实时。
2. 确定指标来源：事件、队列采样、运行时采样或实体状态。
3. 扩展数据结构和 `to_dict()` 输出。
4. 补充缺失数据、空样本、异常顺序的处理。
5. 更新统计面板展示。
6. 更新接口文档。
7. 编写单元测试。

### 14.3 新增配置项

1. 在 `SimulationConfig` 中添加字段和默认值。
2. 在配置弹窗中添加输入控件和边界校验。
3. 在 worker 初始化或业务逻辑中使用该字段。
4. 更新 README 或相关文档。
5. 用至少一个测试或手动验证说明覆盖默认值和边界值。

### 14.4 调整 `frame` 字段

1. 先更新 `docs/realtime_frame_interface.md`，明确字段类型和兼容策略。
2. 修改后端 `_build_frame()`。
3. 修改前端读取逻辑。
4. 更新联调测试方案或示例数据。
5. 运行后端自测并手动启动 UI。

## 15. 当前项目特别注意事项

- 当前 `README.md` 只包含最小运行说明，详细开发约定以本文档为准。
- 当前核心布局仍有较多硬编码，新增空间能力时优先向配置化演进。
- 当前 `served_students` 在业务语义上需要持续和“离场人数 / 完成就餐人数”口径对齐。
- 当前接口文档中部分内容按阶段标记，开发时要区分“已实现”和“计划实现”。
- 当前测试以数据记录和后端自测为主，涉及状态机、路径和 UI 的测试仍需要逐步补齐。
